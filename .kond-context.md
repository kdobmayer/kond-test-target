# Context Bundle for Health Check Endpoint Implementation

## Goal
Add a health check endpoint to the API that returns service uptime.

## 1. Test Command
`npm test` or `jest --coverage --forceExit`

## 2. Build Command
There is no explicit build command. The application is run directly using Node.js: `node src/index.js`.

## 3. Relevant Code Snippets

### Application Setup and Middleware (from `src/app.js`)
The `createApp` function initializes the Express app, applies middleware, and registers the main API routes. The `auth` middleware is applied to `/api` routes.

```javascript
const express = require('express');
const morgan = require('morgan');
const cors = require('cors');
const apiRoutes = require('./routes/api');
const { errorHandler } = require('./middleware/errorHandler');
const { auth } = require('./middleware/auth');
const { initDb } = require('./db');

function createApp() {
    initDb();
    const app = express();

    app.use(morgan('dev'));
    app.use(cors());
    app.use(express.json());
    app.use(auth); // Apply auth middleware to all /api routes

    app.use('/api', apiRoutes);

    app.use(errorHandler); // Centralized error handling

    return app;
}

module.exports = { createApp };
```

## 4. Test Patterns (from `tests/products.test.js`)

### Setup/Teardown for Tests
Tests use `supertest` for making HTTP requests. A test database is set up and torn down for each test suite, and the app is re-created.

```javascript
const request = require('supertest');
const { createApp, setupTestDb, teardownTestDb, seedTestData, API_KEY } = require('./helpers');

let app;

beforeEach(() => {
  setupTestDb();
  app = createApp();
  seedTestData();
});

afterEach(() => {
  teardownTestDb();
});
```

### Making Requests and Assertions
Requests are made using `supertest` and assertions are made on the `status` code and `body` content.

```javascript
describe('GET /api/products', () => {
  it('returns paginated product list', async () => {
    const res = await request(app).get('/api/products').set('x-api-key', API_KEY);
    expect(res.status).toBe(200);
    expect(res.body.data).toHaveLength(3);
    expect(res.body.pagination).toBeDefined();
    expect(res.body.pagination.total).toBe(3);
  });

  it('returns 404 for non-existent product', async () => {
    const res = await request(app).get('/api/products/999').set('x-api-key', API_KEY);
    expect(res.status).toBe(404);
  });
});
```

## 5. Files to Modify/Create

*   `src/controllers/health.js`: Create a new controller file to handle the health check logic.
*   `src/routes/api.js`: Add the new health check route. Alternatively, a separate router could be created for public endpoints, then added to `app.js` before the `auth` middleware. Given the current structure, adding to `api.js` is simpler if the health check needs auth (unlikely) or if it's explicitly excluded from auth. For a public health check, `src/app.js` needs a dedicated public router.
*   `src/app.js`: Import the new health controller and register its route. Consider whether the health check endpoint should be authenticated or not. It should ideally be unauthenticated.
*   `tests/health.test.js`: Create a new test file to cover the health check endpoint.

## 6. Constraints and Gotchas

*   **Authentication:** The `auth` middleware is applied to `/api` routes in `src/app.js`. A health check endpoint (`/health` or `/api/health`) should generally be publicly accessible without authentication. Therefore, the health check route should be registered *before* the `app.use(auth)` middleware in `src/app.js`, or on a separate router that is not affected by the `auth` middleware. The existing `apiRoutes` are mounted at `/api` *after* auth. For an unauthenticated health check, it should likely be on its own route, e.g., `app.get('/health', healthController.getHealth)`.
*   **Response Format:** Adhere to the existing JSON response format, typically with a `data` key for successful responses. For a health check, a simple JSON object containing `status: 'ok'` and `uptime: <seconds>` would be appropriate.
*   **Uptime Calculation:** Use `process.uptime()` for Node.js process uptime.
*   **Error Handling:** While a health check ideally doesn't fail often, ensure any potential errors (e.g., if it were to check a database connection) are handled gracefully and return an appropriate HTTP status code and JSON error format. For a simple uptime, it's less critical.
*   **Database Interaction:** A simple uptime health check should *not* interact with the database to keep it lightweight and fast, unless specifically requested to check database connectivity as part of "health".
